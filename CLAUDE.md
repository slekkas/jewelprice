# JewelPrice Pro — Project Context

## What this is
JewelPrice Pro is a bilingual (Greek/English) jewelry pricing Progressive Web App for Lekkas Jewelry, a seasonal jewelry store in Karpathos, Greece. It is used daily in the store by the owner (Sotiris) and an employee.

## Critical technical constraints — never violate these
- The entire app lives in a **single file: `jewelprice.htm`**. Do NOT split it into multiple files, and do NOT create or edit `index.htm` (the file was renamed from index.htm; old references to that name are outdated).
- Hosted on **GitHub Pages**. The app is accessed via its full URL ending in `/jewelprice.htm` — never break or rename this path, as users have it bookmarked/installed.
- Backend: **Firebase Realtime Database** (migrated from JSONBin and jsonstorage.net — do not reintroduce those).
- All UI text must exist in **both Greek and English**, following the app's existing i18n pattern.
- Greek characters CANNOT be encoded in QR codes generated for BarTender UltraLite (SHIFT-JIS limitation). Label codes therefore use parallel Greek/Latin codes.
- QR camera scanning prefers the browser's native **BarcodeDetector** and falls back to an **inlined jsQR** decoder (in a `<script id="jsqr-lib">` block) for desktop browsers that lack it (Chrome on Windows/Linux). `canDecodeQr()` is true where QR decoding works and false on Safari/iOS.
- **Text (OCR) scanning** (v10.6): a "Scan code text" toggle inside the scanner overlay (`toggleScanMode`) reads the **printed reference code** (e.g. `PE0142`, from `formatSeqId`) instead of the QR square — the fallback for sun-faded/scratched tags that killed the old attempt (which tried to OCR the free-form Greek code `ΚΑΛ18Σ4.5`). Uses **Tesseract.js**, lazy-loaded from CDN on first use (`loadTesseract`) and cached in IndexedDB (needs internet the first time only). The engine runs with a narrow char whitelist (`BCEGHKLNOPRTWX0123456789`) + single-line PSM; `extractRefCode` pulls a `[A-Z]{2}\d{4}` token, applies position-aware confusion fixes (O↔0, B↔8, etc.) and validates the 2-letter prefix against `REF_PREFIXES`. The live path crops a **wide, short strip** across the centre whose height scales with the frame **width** (not height) so it stays a tight one-line strip in any orientation — a fixed fraction of frame *height* produced a tall, sparse crop on portrait phones that single-line OCR couldn't read — and it OCRs that strip at a few normalised heights, taking the first valid read (text size varies with how close the tag is held); gallery/photo OCRs the whole image. The live loop accepts a code as soon as one frame reads a valid token that **matches a real inventory item** (`refCodeExists`) — the existence check rejects misreads into non-existent codes, and `lookupCode` then shows the item so a wrong read is caught by eye. (An earlier two-frame agreement gate was removed in v10.7: it caused a ~2s delay and could get stuck showing "Reading…" forever if the user moved the tag away before the second confirming frame.) Text mode shows a dashed **read-zone guide band** (`.qr-textguide`) marking where to place the code, and the frame turns **green** (`.qr-video-wrap.aligned`) the moment a valid code is readable — a live "aligned, hold still" signal driven by OCR success itself (no tilt sensor; readability *is* alignment). Result is fed to `handleScannedCode` → `lookupCode`, which already resolves ref codes to items. Because the Calc scanner now always has a working path (QR where possible, else OCR), the Calc scan icon **shows on all devices including iOS/Safari** — where it opens straight into text mode. **Inventory scanning stays QR-only** (its confirm flow parses QR, not ref codes), so the text toggle is hidden there.
- Users are on mobile: employee uses a Xiaomi Redmi (Android/Chrome), another user is on iPhone (Safari). Test logic must account for both.

- **Stone-system review & hardening (v10.15):** three round-trip bugs fixed. (1) **Missing abbreviations** — `Akoya Pearl`, `Freshwater Pearl`, `South Sea Pearl`, `Tahitian Pearl` and `Jade` were in `STONE_TYPES` but absent from `STONE_ABBREV_MAP`, so `stoneToLabelPart` encoded them by full name (with carat → re-scanned as **Diamond**) or `type.slice(0,4)` (no carat → dropped on scan). Added them to `STONE_ABBREV_MAP` (keys `AKOY`/`FRES`/`SOUT`/`TAHI`/`JADE`, matching the historical 4-letter fallback so already-printed no-carat tags now parse) and to `COLORED_FULL`/`COLORED_FULL_EARLY`. (2) **Scan-to-inventory dropped qty/melee** — the invScan item builder hardcoded `qty: 1`, so scanning a `4xRUB`/`tZIR` tag into inventory lost the count/melee; now uses `s.qty || 1`. (3) **Label-save `stonesDesc` from lossy QR** — it rebuilt the description from `parseCodeInput(fullQR)` and merged DOM shape/price **by index**, so a no-carat Diamond (absent from the QR) was dropped and, when any stone was skipped in the code, shape/price shifted onto the wrong stone. Now builds `stonesDesc` directly from the DOM stone rows (the complete source the QR was itself built from), gated by the same stone-section-open check as `buildLabelStrings`, falling back to the QR only when there are no DOM rows. Note: 3 divergent copies of the diamond/colored price-per-carat tables still exist (calc estimate / inv-edit live price / another) — a maintainability risk, left as-is.
- **Photo-only edits don't trigger a tag reprint (v10.14):** the Edit Item reprint prompt now fires only when the **stones actually change** in that edit session, not whenever the rebuilt suffix differs from the stored code. On open, `invEdit` captures a baseline stone suffix (`_invEditStoneRowsEl.dataset.baselineSuffix = _buildStoneQRSuffix(...)`); on save it flags the tag stale only if `_newSuffix !== baseline`. This stops photo-only edits (e.g. the bulk HD-photo re-upload) from nagging for a reprint, including on legacy items whose stored code is malformed (those `+`-form codes stay in the DB and the tolerant parser handles them; editing/reprinting still auto-corrects them when the user does change stones).
- **Stone-code separator fix — skipped leading stone (v10.13):** the stone suffix on a code uses `-` before the FIRST stone and `+` before each subsequent one. The generators (`buildLabelStrings`, `_buildStoneQRSuffix`) previously chose the separator from the stone's **array index**, but a stone can be skipped (e.g. a carat-less Diamond yields no label part via `stoneToLabelPart`). So an item whose first stone was a carat-less Diamond followed by e.g. 4× Ruby produced `KAL18S1.3+4xRUB` (no `-`) instead of `KAL18S1.3-4xRUB`; with no `-`, `parseCodeInput` couldn't split base from stones and the calc showed "Code not found". Fixed by tracking the count of parts **actually emitted** (not the index) in all three generator loops. `parseCodeInput` is now also tolerant of the malformed form (no `-` but a `+` → treat the first `+` as the stone boundary) so already-printed bad tags still scan; and the edit-save + restock QR-regen paths normalise a leading `+` suffix to `-`, so editing/reprinting such an item auto-corrects its code. Existing bad codes in the DB are left as-is (the tolerant parser handles them).
- **Stone melee marking recovered from the QR code on edit (v10.12):** the printed QR code is the authoritative source for a stone's quantity/melee (`t` prefix, e.g. `-tZIR`). Older items whose stored `stonesDesc` lost the "melee" word (built before the recent qty change) used to re-parse as plain qty=1 in the Edit Item form, so simply changing a photo regenerated the code as `ZIR` and forced a spurious "print new tag" prompt. `invEdit` now overlays the qty read from `parseCodeInput(item.qrCode).stones` onto the `stonesDesc`-parsed rows (matched by index; or rebuilds rows straight from the code if `stonesDesc` yields none), so melee is restored on load and, on save, `buildStoneDescription` self-heals the description to include "melee" and the regenerated suffix matches the old one (no reprint prompt).
- **Restock doesn't inflate the gain % (v10.17):** the bulk-chain "+ Restock" flow (`confirmBulkRestock`) used to add grams to an item's weight without touching `initialRetail`, so the newly added stock made the current retail jump while the initial price stayed flat — the "Initial price (+x%)" gain line then ballooned (e.g. adding 57g of 14K chain showed a fake +32% rise). It now measures retail at the current gold price *before* and *after* adding the grams (gold price is identical in that instant, so the whole delta is the value of the new metal) and credits that delta into `initialRetail`, so the gain % reflects only real gold-price movement, not how much stock is on hand.
- **Gain sort & filter (v10.17):** to spot such anomalies fast, the inventory Sort menu gains **"Gain: High → Low"** / **"Gain: Low → High"** (`gain_high`/`gain_low`) and the Filter sheet gains a **"Price Gain"** group with **> +3% / +5% / +10%** single-select threshold pills (`_invFilters.gainMin`, a number; 0 = off). Both are **owner/manager only** (hidden from employees, mirroring the appreciation line). A shared helper `invGainPct(item)` computes an item's gain vs its `initialRetail` (returns `null` for Other-type, sold, or baseline-less items, which sort to the bottom and are excluded by the filter).
- **No silent stone auto-pricing (v10.19):** `renderInvList` previously ran a background self-heal that, for any stone item whose `stoneCost` was left blank (null/undefined — never `> 0` or explicit `0`, which it skipped), re-parsed the QR, looked the stone up in the shared stone-price cache (`lookupStonePriceCache`), and **silently wrote that cached price onto the item and persisted it to Firebase**. A blank stone price now simply counts as €0; stones are priced only by explicit user action (manual entry or the ✨ Estimate button in Edit/Scan, both tap-only — neither auto-applies on open). - **Reference code frozen to the printed tag (v10.20):** the printed ref code (e.g. `NK0342`) was previously *derived live* from `jewelryType` everywhere it appeared (`formatSeqId(seqId, jewelryType)`), so editing an item's type silently rewrote the code across the app, the in-app search, the OCR text-scan lookup **and** the public website — leaving the physical tag (and customers who typed it into the showroom search) pointing at a code that no longer resolved. The prefix is now "frozen" via a stored `item.refType` field the first moment the type is changed in `invEdit` (captures the *previous* type before overwriting), and every display/lookup goes through a new helper `itemRefCode(item)` = `formatSeqId(item.seqId, item.refType || item.jewelryType)`. The new-item label-creation paths (`formatSeqId(_pendingSeqId, …)`) are unchanged (a fresh item's type == its code).
- **Android back-gesture guard (v10.22):** on gesture-navigation phones (e.g. Pixel) the edge back-swipe used to exit the installed PWA in one stroke — because switching tabs / opening modals never pushed browser-history entries, so the first "back" popped the app's only entry and closed it. **The reliable design makes the browser history mirror the tab path proactively:** `_jpTrackPanel` (called from the top of `showPanel`) pushes one *real* history entry on every genuine forward tab change and remembers the tab it came from in `_jpFrames`; a single permanent home-floor guard is pushed once at startup. A back gesture (`popstate` → `_jpHandleBack`) then just *consumes* a pre-existing entry — the code never re-pushes history during the common back paths (re-pushing inside `popstate` is unreliable on Android Chrome and was why the app used to close with no warning). `_jpFrames.length` is the source of truth for depth (not `history.state`), so: (1) if a full-screen overlay/modal is open it's closed the way the user would — `_jpTopOverlay()` finds the topmost visible `position:fixed` element at `z-index ≥ 50000` (real modals are ≥ 99996; tab panels are z:100) and `_jpCloseOverlay()` clicks its own Cancel/close control (skipping destructive/confirming actions), falling back to backdrop-click / hide-or-remove, then re-pushes one entry to keep depth in sync (modals have no frame of their own); (2) else if tab frames remain, it pops one and returns to that tab (no re-push); (3) at the home floor (no frames) a first swipe shows a bilingual "Swipe again to exit" toast (`_jpShowExitToast`) and arms a ~3s window, and only the second swipe calls `history.back()` to actually close the app — and this degrades gracefully (if the re-push ever fails, the confirming swipe still exits). **Calc is a hard home floor:** arriving at Calc (tap or programmatic `showPanel('calc')`) collapses any tab frames back to the floor via `history.go(-n)` (the resulting pops are swallowed with the `_jpUnwind` counter, self-healing after 700ms), so a back gesture from Calc ALWAYS warns rather than walking forward into a tab you tapped through. **Resume re-arm:** a PWA has no API to force-quit itself, so `history.back()` only backgrounds the app on Android; when it returns to the foreground the same page resumes with its history stack emptied, which would let the next back close it with no warning — so `_jpResumeRearm` (on `visibilitychange`→visible, `pageshow` and `focus`) re-pushes a fresh home guard whenever we come back after an exit attempt (`_jpExited`). **Double-fire dwell guard:** Android's edge back-swipe often fires `popstate` twice for a single gesture; at the home floor that used to arm on the first fire and immediately exit on the second, and on other tabs it silently double-stepped. The exit branch requires ≥500 ms since the arming swipe (`_jpArmedAt`) before `history.back()` runs, so a single gesture can only ever ARM; only a deliberate later swipe exits. **Guard must be pushed inside a user gesture (Chrome history-manipulation intervention):** this was *the* real cause of "open on Calc, swipe back → immediate close, no warning". Chrome (≥74) on Android marks any `pushState` made **without a prior user activation** as *skippable*, so a guard pushed at page load is skipped by the back gesture and the PWA closes with **no `popstate`** (so none of the above logic runs). The guard is therefore established from the **first real user gesture** — `_jpEstablishGuard`, wired to capture-phase `pointerdown`/`touchstart`/`keydown`/`click` listeners added in `_jpInitBackGuard` — where Chrome honours it. `_jpGuardUp` makes it idempotent; `_jpResumeRearm` resets `_jpGuardUp=false` so the next gesture rebuilds it after a resume. **Residual limitation:** if the user opens the app and swipes back having touched *nothing* at all (no tap/scroll/keypress), there is no user activation, so Chrome still closes it — unavoidable by design (the same activation requirement also suppresses `beforeunload`), and harmless since no work exists to lose. Any interaction at all (including a scroll's `touchstart`) arms the guard. The guard is enabled only in standalone/installed PWAs or on coarse-pointer (touch) devices, so a desktop browser's Back button is untouched. It also persists the last active tab (`localStorage 'jpro_lastPanel'`) and, on open, reopens it (spirit of "continue where I was") with one seeded `calc` frame so back still returns to Calc first, unless following a `#item=` deep link. Init: `_jpInitBackGuard()` from the boot `DOMContentLoaded` after `checkDeepLinkHash()`.
- **Inventory list performance — instant open + windowing (v10.23):** the Stock tab was the only tab that stalled on open, and the cost grew with the item count (382 → 1000+). Three fixes, all read/render-only (no data writes, no behaviour change): (1) **Instant render from cache + background refresh** — `renderInventory()` now draws immediately from the in-memory `_inventoryCache` (like every other tab) whenever it's already populated, then re-fetches from Firebase in the background and only re-renders if the JSON actually changed *and* the panel isn't scrolled down (`panel.scrollTop < 200`), so a cross-device change still lands without yanking the list. (3) **Windowed rendering** — instead of building all N cards at once, `renderInvList` stores the full sorted rows in `_invSortedRows` + a `buildCard` closure in `_invBuildCard`, then reveals `INV_PAGE_SIZE` (50) at a time: `_invRenderNextPage()` appends the next page (via `insertAdjacentHTML`), the panel's scroll listener (in `_initInvScrollTop`) calls it as you near the bottom, and `_invMaybeFill()` tops up until the viewport overflows (guarded to no-op when the panel is hidden, `clientHeight===0`). "Jump to card" flows (`_jumpToInvCard`, `handleDeepLinkItem`, the sale→item jump) call `_invEnsureRendered(id)` to page up to a target that isn't in the DOM yet; `invSelectAll` selects every id in `_invSortedRows` (not just rendered cards); newly appended cards re-apply select-mode via `_invRestoreSelectMode`. Measured ~13–16 ms to render at 1000 items with only 50 cards in the DOM.

  - **€20 floor** (`state.autoRoundMin`), chosen by the owner: nearest-5 moves a price by up to €2.50, which is nothing on a €239 ring and +56% on a €3.20 clasp. Below the floor the exact price stands.
  - **Self-service, per the owner's standing preference** — he asked to "test it for a few days and revert if I don't like it", so instead of a temporary experiment there is a permanent control: **Settings → Tax & Margin** has a step select (Off / €1 / €5 / €10, default **€5**) and the floor (default **€20**). `autoRoundStep`/`autoRoundMin` are in `cloudConfig`/`applyCloudConfig`, so a change reaches every device. Reverting is picking "Off"; no code change, no deploy.
  - **Defaults are read through `_autoRoundStep()`/`_autoRoundMin()`, never off `state` directly** — an existing device's saved `jpro_state` predates these keys, so a bare `state.autoRoundStep` would be `undefined` (verified: the helpers fall back to 5/€20 and such a device rounds correctly).

## Public website & data security
Alongside the private pricing app (`jewelprice.htm`) the repo also serves a small **public website** on the same GitHub Pages site:
- `index.html` — public landing page (hero, hours, contact, links to the showroom).
- `showroom.htm` — password-gated "Private Showroom": customers browse in-stock pieces (photo, description, type, metal, stones), select favourites, and send an inquiry via EmailJS. No prices shown. The item lightbox has **image zoom** (`_lbInitZoom`, called from `openLightbox`): desktop (fine pointer + hover) hovers to zoom and moves the cursor to pan; touch taps to zoom in centred on the tap, drags to pan, taps again to reset — for inspecting tiny pieces like children's bracelets. Purely `transform: scale()` + `transform-origin` on `.lb-img` inside an `overflow:hidden` `.lb-zoom` wrapper; showroom-only, no `APP_VERSION` bump.
- `item.htm` — single-item "view item" page opened from a QR/`?qr=` link (used in inquiry emails and physical tags). **Bilingual (EL/EN)** with a header language toggle: reads `localStorage 'jpro_lang'` (else browser language), persists on toggle, and re-renders the loaded item (`_item`/`_qr`) in the chosen language via `T`/`METAL_L`/`GENDER_L`/`TYPE_L` maps + `setLang(l)`. Reads `descEl`/`descEn` from the public feed. Jewelry-type labels mirror `INV_JEWELRY_TYPES`.
- `quote.htm` — standalone quote view. **Effectively unused:** quotes/certificates/orders/catalogs are all delivered as a **PDF** (`shareQuoteAsPDF`); the `viewerUrl` built in `shareQuote` is never shared (both `openQuoteWindow` callers pass `null`) and serves only as an "is Firebase configured?" guard for the history save. The DB secret was removed from that URL in v11.36 — an unused URL is no place for a master key — but this fixed no real leak, and **`/jewelprice_quotes_html` must stay PRIVATE** (a briefly-published world-readable rule was reverted the same day; it exposed stored quote HTML for a link that does not exist). Removing the dead quote-link machinery is a pending cleanup — see `SECURITY_PASS.md`. The quote **content** is baked HTML already in the language it was created in; only the viewer **chrome** (loading/error text, Share/Save-PDF buttons) is translated, auto-detected from `localStorage 'jpro_lang'`/browser language (`QT`/`qt()`, no toggle — the content language is fixed). `buildFallbackCard` is dead code.
- `changelog.htm` — standalone bilingual version history (v10.66+). Static — the changelog data is embedded in the page; reads no Firebase node, so it has no data-security surface.
- `manual.htm` — standalone bilingual User Manual (v10.68+). Static accordion; content embedded in the page (`MANUAL` object); reads no Firebase node.

**NEVER put a key, token or secret in `jewelprice.htm` (v11.85).** The file is served publicly from GitHub Pages — anyone can read every byte of it. The OpenRouter API key was hardcoded in **three** places as `atob('c2stb3It…')`; base64 is an *encoding*, not encryption, so the key was readable (and spendable) by anyone who opened the page source, and it was live on the public site. It now lives in the private Firebase config under `aiKey`, entered once via the **Sync tab → OpenRouter API Key** field (`saveAiKey`, which validates against `GET /api/v1/key` before storing, so a mistyped or revoked key can't be saved silently) and read at call time through the single helper **`_aiKey()`**. The three call sites `throw new Error('no-ai-key')` when it is absent, which their existing `catch` turns into the ordinary local-table fallback. Same pattern as the Telegram bot token. **If a future feature needs a credential, it goes in the config — never in the file.**

**Security model (important — do not regress):**
- These public pages must **never** carry the Firebase legacy **master DB secret** (it grants full read/write to the entire database) and must **never** read the private `/jewelprice_inventory` node (it contains cost, margin, vendor and supplier fields).
- Instead the app maintains a separate, **display-only public mirror** at Firebase node **`/jewelprice_public`**, containing only the safe fields in `PUBLIC_SAFE_FIELDS` (qrCode, descriptions, jewelryType, refType, metal, karat, goldColor, weight, gender, alsoGenders, handmade, stonesDesc, seqId, photoUrl, photoUrlHi, status, dateAdded). `jewelprice.htm` keeps it in sync automatically via `syncPublicItem` / `syncPublicDelete` (on every inventory save/delete) and a once-per-session self-healing `republishPublicShowroom()` (called from `renderInvList`). Only in-stock, non-bulk items are published (`publicProjection`).
- **Firebase Realtime Database rules** must keep everything private by default and expose **only** the two display nodes as world-readable, read-only:
  ```json
  { "rules": { ".read": false, ".write": false,
      "jewelprice_public": { ".read": true, ".write": false },
      "jewelprice_showroom_config": { ".read": true, ".write": false } } }
  ```
  (The app itself still reads/writes everything because it authenticates with the DB secret, which bypasses rules.) `jewelprice_showroom_config` was added in v12.04 and holds **only** `{enabled, hash}` for the showroom gate — never the readable code. The canonical copy of this JSON in code is `_srRulesSnippet()`; keep the two in step.
- `showroom.htm` and `item.htm` read `/jewelprice_public` **without any auth token**. If either ever needs the secret again, that's a security regression.
- Changes to the public website files do **not** bump `APP_VERSION`/changelog — that constant governs the pricing app (`jewelprice.htm`) and its in-app version banner only.
